# Refact packaging

These templates distribute the standalone `refact` binary without creating a
second build pipeline. They are inert in repository CI and consume the assets
published by `.github/workflows/agent_engine_release.yml` for an
`engine/v<version>` tag.

## Release contract

Every asset URL has this shape:

```text
https://github.com/JegernOUTT/refact/releases/download/engine/v<version>/refact-<version>-<target>.<extension>
```

Windows targets use `.zip`; Unix targets use `.tar.gz`. Each archive contains
only `refact` or `refact.exe` at its root and has an adjacent `<archive>.sha256`
sidecar. `tools/dev/package_engine_release_assets.sh` is the authoritative
packager.

Current release targets are:

| Platform | Target | Extension |
| --- | --- | --- |
| Windows x64 | `x86_64-pc-windows-msvc` | `.zip` |
| Windows x86 | `i686-pc-windows-msvc` | `.zip` |
| Windows ARM64 | `aarch64-pc-windows-msvc` | `.zip` |
| Linux x64 | `x86_64-unknown-linux-gnu` | `.tar.gz` |
| Linux ARM64 | `aarch64-unknown-linux-gnu` | `.tar.gz` |
| macOS ARM64 | `aarch64-apple-darwin` | `.tar.gz` |

The Homebrew and npm templates also encode the contract name for
`x86_64-apple-darwin`. That target must be re-enabled in
`agent_engine_build.yml` before publishing either channel with Intel macOS
support.

## Version and checksum stamping

All manifests use the exact `__REFACT_VERSION__` token. Before publishing,
replace it with the engine release version without the `v` prefix. The checksum
tokens use `__REFACT_SHA256_<TARGET>__`, where punctuation in the Rust target is
replaced by underscores and letters are uppercased. Replace each checksum token
with the first field of the corresponding `.sha256` sidecar.

This convention is intentionally simple so `tools/bump_release_version.py` can
be extended later. This card does not change that release script or enable
packaging jobs in CI.

## Homebrew

1. Stamp `homebrew/refact.rb` with the version and Unix archive checksums.
2. Run `ruby -c homebrew/refact.rb` and `brew audit --strict refact` in a tap.
3. Commit the formula to the Homebrew tap and test `brew install refact` followed
   by `refact version` on each published platform.

Intel macOS publishing is blocked while the corresponding engine build target
is disabled. Linux x64/ARM64 and macOS ARM64 assets are currently produced.

## winget

1. Stamp all files under `winget/` with one version and the three Windows archive
   checksums.
2. Run `winget validate --manifest winget`.
3. Submit the version directory to `microsoft/winget-pkgs` using `wingetcreate` or
   a normal pull request.

## Scoop

1. Stamp `scoop/refact.json`, then validate it with
   `python3 -m json.tool scoop/refact.json`.
2. Commit it as `bucket/refact.json` in the Scoop bucket.
3. Future versions can be generated by Scoop's `checkver`/`autoupdate`; hashes
   come from each release archive's `.sha256` URL.

## npm

1. Stamp `npm/package.json` and its installer URLs with the release version.
2. From `npm/`, run `npm test`, `node postinstall.js --dry-run`, and
   `npm pack --dry-run`.
3. Publish with `npm publish --access public`.

The dependency-free postinstall script selects a release archive, verifies its
sidecar, and installs the native binary behind the package's `refact` launcher.
`REFACT_RELEASE_BASE_URL` supports mirrors with the same tag and asset layout.

## Docker

The Dockerfile fetches a release selected by `REFACT_VERSION`, verifies its
sidecar, and runs `refact daemon --foreground`. See `docker/README.md` and
`docker/compose.yaml` for build, project-mount, and authenticated LAN examples.

## Automation follow-up

No package registry credentials or external repositories are configured here,
so repository CI remains unchanged. A later release-automation card can add an
opt-in job after `agent_engine_release.yml` publishes assets to stamp templates,
open Homebrew/Scoop/winget pull requests, and publish npm. Until those credentials
and destinations are approved, use the manual steps above.
