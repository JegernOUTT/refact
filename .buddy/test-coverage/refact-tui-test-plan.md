## Proposed test additions

### 1. `src/render/mod.rs`

Append a `#[cfg(test)] mod tests` block with tests that exercise the env-var
branches of `color_enabled_from_env`. Use `std::env::set_var` / `remove_var`
inside the test (these are unit tests, not run in parallel with the rest of
the crate's env-var-sensitive code, so there is no contention). Cases:

- `NO_COLOR` unset, `TERM` unset -> `true`.
- `NO_COLOR` set to anything (even empty) -> `false`.
- `NO_COLOR` unset, `TERM=dumb` -> `false`.
- `NO_COLOR` unset, `TERM=xterm-256color` -> `true`.

### 2. `src/streaming/commit_tick.rs`

Append a `#[cfg(test)] mod tests` block. Both items in this file are
`pub(crate)`, so the test module must live in the same crate root module
(inside `commit_tick.rs`) and use `super::*` plus `super::super::chunking::*`.
Cases:

- `drain_with_policy` with a fresh `AdaptiveChunkingPolicy` and
  `queued_lines = 3` runs the `Smooth` plan: the closure is called exactly
  once with `DrainPlan::Single` and its return value propagates.
- `drain_with_policy` with `queued_lines = 8` runs the `CatchUp` plan: the
  closure is called exactly once with `DrainPlan::Batch(8)` and the returned
  `Option<String>` is forwarded.
- `drain_with_policy` with `queued_lines = 0` calls the closure once and
  forwards `None` when the closure returns `None` (verifies the `None` path
  is not silently dropped).

The trivial `run_commit_tick` wrapper can be covered indirectly by the
existing `controller.rs` integration tests, so it does not need its own
assertion.
